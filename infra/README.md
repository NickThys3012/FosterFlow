# FosterFlow Infrastructure (Bicep)

Infrastructure-as-Code for FosterFlow's Azure production environment. Covers Epic 0 issues:

| Issue | Story | What this provisions |
|-------|-------|----------------------|
| #37 | US-INF-1.1 Resource Group & Basics | `rg-fosterflow-prod-swe` (swedencentral), `stfosterflowprodswe`, `kv-fosterflow-prod-swe` + secrets, tags |
| #38 | US-INF-1.2 Azure SQL | `sql-fosterflow-prod-swe` + `sqldb-fosterflow-prod-swe` (Basic), firewall, 7-day backups |
| #39 | US-INF-1.3 Blob Storage | `cat-photos` private container in the storage account |
| #40 | US-INF-1.4 App Service | `asp-fosterflow-prod-swe` (B1 Linux) + `app-fosterflow-prod-swe` with Key Vault references |

> **Naming**: resources follow the [Microsoft Cloud Adoption Framework](https://learn.microsoft.com/azure/cloud-adoption-framework/ready/azure-best-practices/resource-naming)
> convention `<type-abbreviation>-<workload>-<env>-<region>` (`rg`, `st`, `kv`, `sql`, `sqldb`, `asp`, `app`;
> storage accounts omit hyphens). Names are derived from the `workload` / `environmentAbbreviation` /
> `regionAbbreviation` parameters, so changing those tokens renames everything consistently.

## Layout

```
infra/
├─ main.bicep                 # subscription scope: creates the RG, then deploys resources.bicep
├─ resources.bicep            # resource-group scope: wires all modules together
├─ modules/
│  ├─ storage.bicep           # storage account + private cat-photos container (#37/#39)
│  ├─ sql.bicep               # SQL server + database + firewall + backups (#38)
│  ├─ keyvault.bicep          # Key Vault + 4 secrets, builds connection strings (#37)
│  └─ appservice.bicep        # plan + web app + KV role assignment (#40)
├─ main.example.bicepparam    # copy to main.bicepparam and fill in
└─ deploy.sh                  # what-if + deploy convenience script
```

## What gets created

- **Tags** on every resource: `project=fosterflow`, `environment=production`, `hackathon=2026`.
- **Storage**: `stfosterflowprodswe` (StorageV2, Standard LRS, TLS 1.2, blob public access **disabled**) with a
  private `cat-photos` container. Private access means uploads/reads use SAS tokens generated by the API.
- **Key Vault**: `kv-fosterflow-prod-swe` (RBAC data-plane) with secrets `JwtSecret`, `DbConnectionString`,
  `BlobStorageConnectionString`, `LokiUrl`. The DB and blob connection strings are assembled inside the vault
  module so the raw SQL password and storage key never appear in deployment outputs.
- **SQL**: `sql-fosterflow-prod-swe` + `sqldb-fosterflow-prod-swe` (Basic tier, local backup redundancy, 7-day PITR). Firewall
  allows Azure services (`0.0.0.0`) and, optionally, one developer IP.
- **App Service**: `asp-fosterflow-prod-swe` (B1 Linux) + `app-fosterflow-prod-swe` with a system-assigned identity, **Always
  On**, **HTTPS-only**, health check at `/health`, and app settings that reference Key Vault secrets. The
  identity is granted the **Key Vault Secrets User** role on the vault.

App settings use double-underscore names so .NET config binds them: `ConnectionStrings__Database`,
`ConnectionStrings__BlobStorage`, `Jwt__Secret`, `Jwt__Issuer`, `Jwt__Audience`, `Loki__Url`,
`ASPNETCORE_ENVIRONMENT=Production`. (`ConnectionStrings__Database` matches the name the API actually reads
via `GetConnectionString("Database")`.)

## Deploy

```bash
az login
az account set --subscription <your-sub-id>

export SQL_ADMIN_PASSWORD='<16+ char strong password>'
export JWT_SECRET='<32+ char random string>'
export LOKI_URL='https://your-loki-endpoint'      # optional
export CLIENT_IP="$(curl -s -4 ifconfig.me)"       # optional, for local SQL access (must be IPv4)

cd infra
./deploy.sh
```

Or manually:

```bash
az deployment sub create \
  --location swedencentral \
  --template-file main.bicep \
  --parameters sqlAdminPassword=$SQL_ADMIN_PASSWORD jwtSecret=$JWT_SECRET
```

## Database migrations (#38)

Migrations are applied **automatically by the API at startup** (`MigrateDatabaseAsync()`
in `Program.cs`), so a freshly provisioned database gets its schema on first boot — no
manual step is required after a normal deploy.

If you ever need to apply migrations manually (e.g. from a developer machine whose IP is
whitelisted via `allowedClientIp`), run from the solution dir (`FosterFlow/FosterFlow`):

```bash
dotnet ef database update \
  --project FosterFlow.Infrastructure --startup-project FosterFlow.Api \
  --connection "$(az keyvault secret show --vault-name kv-fosterflow-prod-swe --name DbConnectionString --query value -o tsv)"
```

## Notes & caveats

- **Region**: defaults to **swedencentral**. Issues specify `uksouth`, but this subscription has **no B1 App
  Service VM quota** in uksouth (`SubscriptionIsOverQuotaForSku`), so the default was moved to swedencentral
  where `what-if` validated cleanly. Override `location` once uksouth quota is granted.
- **Loki / monitoring is optional**: leave `lokiUrl` empty to deploy now. The `LokiUrl` Key Vault secret and
  the `Loki__Url` app setting are only created when `lokiUrl` is provided (Key Vault rejects empty secret
  values). Add it later without redeploying everything:
  ```bash
  az keyvault secret set --vault-name kv-fosterflow-prod-swe --name LokiUrl --value "https://your-loki"
  az webapp config appsettings set --name app-fosterflow-prod-swe --resource-group rg-fosterflow-prod-swe \
    --settings "Loki__Url=@Microsoft.KeyVault(VaultName=kv-fosterflow-prod-swe;SecretName=LokiUrl)"
  ```
  Or just re-run the deployment with `lokiUrl` set.
- **Globally-unique names**: `storageAccountName`, `keyVaultName`, `sqlServerName`, and `appServiceName` must
  be unique across Azure. Override any that collide via parameters.
- **Runtime version**: the repo targets **.NET 10** (`net10.0`), so `linuxFxVersion` defaults to
  `DOTNETCORE|10.0` rather than the `.NET 8` named in issue #40. Override `linuxFxVersion` if your App Service
  region doesn't yet offer .NET 10.
- **SQL auto-pause**: issue #38 mentions auto-pause, which is a *serverless* (General Purpose) feature. The
  **Basic** tier specified for cost does not auto-pause; switch to GP serverless if auto-pause is required.
- Application code (Blazor WASM publish into `wwwroot/`, SAS generation, etc.) is deployed/handled separately;
  this folder only provisions the infrastructure.
